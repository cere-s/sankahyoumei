# CLAUDE.md

## Project Overview

このプロジェクトは、コスプレイベント向けの参加表明サイトです。

サービスの中心価値は、イベント前に「誰に会えるか」「何を撮れるか」「誰がどの作品・キャラで参加するか」を見える化することです。

単なるイベント一覧サイトではなく、イベント前の期待感・交流・撮影予定・SNS共有を支援するサービスです。

## Product Concept

コンセプト:

- 好きでつながる、コスプレ参加表明サイト
- イベント前に、会える人と撮れるものを見つける
- 参加者・作品・キャラ・撮影予定が見える場所

このサービスでは、以下を重視します。

1. イベントが魅力的に見えること
2. 参加者が魅力的に見えること
3. 登録した人が「登録した甲斐があった」と感じること
4. 未登録ユーザーが「自分も登録したい」と思えること
5. SNSで共有した時に見栄えが良いこと
6. スマホで迷わず登録・閲覧できること

## Main Users

### コスプレ参加者

- 作品名、キャラ名、衣装、参加時間帯を表明したい
- 同じ作品の人や会いたい人を探したい
- 撮られたい、交流したい、挨拶されたいなどの温度感を出したい
- Xで自分の参加表明を共有したい

### カメラマン

- 撮りたい作品、キャラ、衣装傾向を表明したい
- 当日撮れそうな被写体をイベント前に探したい
- ポートフォリオやSNSを見てもらいたい
- 撮影予定や交流のきっかけを作りたい

### 一般参加・未定の人

- とりあえず参加することを表明したい
- 会いたい人、気になる作品、参加者の傾向を確認したい

## UX Principles

### Mobile First

このサイトはスマホ利用を最優先に設計してください。

- 片手で見やすい
- フォーム入力が重すぎない
- カードの情報が一目で分かる
- SNSアプリ内ブラウザでも崩れない
- 重要CTAが迷子にならない

### 登録フォーム

登録フォームは、入力負荷を下げることを重視してください。

望ましい設計:

- ステップ式フォーム
- 必須項目は最小限
- 任意項目は後から編集できる
- コスプレ参加者とカメラマンで入力項目を出し分ける
- 1イベントで複数キャラ登録できる
- カメラマンは撮りたい作品・キャラを複数登録できる

### 参加者カード

参加者カードは、単なるデータ表示ではなく「この人に会いたい」「撮りたい」「見つけたい」と思える見た目にしてください。

優先して見せる情報:

- 表示名
- Xユーザー名
- 参加種別
- 作品名
- キャラ名
- 衣装
- 参加時間帯
- 撮影・交流の温度感
- プロフィールURL / ポートフォリオURL
- 予定メモ

## Feature Priority

機能追加時は、以下の優先度で判断してください。

### Priority A

最優先で実装・改善するもの。

- イベント詳細ページ
- 参加表明作成・編集
- 参加表明一覧
- 作品名・キャラ名での視認性向上
- 参加種別ごとの表示
- X共有導線
- OGP最適化
- スマホUI改善
- 管理者向けの基本的な確認機能

### Priority B

Aが安定してから検討するもの。

- 撮りたい / 撮られたい / 交流したい の意思表示
- 同じ作品・キャラの人を見つけやすくする導線
- 人気作品・参加人数ランキング
- FAQ
- 簡易分析
- 運営のみが見られるテスト機能

### Priority C

慎重に検討するもの。

- DM機能
- 通知機能
- フォロー機能
- 複雑なレコメンド
- 汎用SNS化する機能

DMや通知など、ユーザー間トラブル・通報・ブロック・法務・運用負荷が増える機能は安易に実装しないでください。

## Design Direction

デザインは、明るくポップで少し近未来的なWebサービス感を目指します。

ただし、単なるキラキラ装飾にしないでください。

重視すること:

- イベント前の高揚感
- SNSで人が集まっていく感じ
- 参加者カードの見やすさ
- 作品名・キャラ名の読みやすさ
- 登録したくなる期待感
- スマホでの情報密度
- CTAの分かりやすさ

避けること:

- 装飾過多
- 背景がうるさくて文字が読みにくい
- ただのアイドル系・魔法少女系デザインに寄りすぎる
- 汎用イベントサイトのような硬い見た目
- 管理画面のような無機質な見た目

## Copywriting Rules

文言は、コスプレイベント参加者に自然に伝わる言葉を使ってください。

良い方向性:

- イベント前に、会える人を見つけよう
- 参加予定の作品・キャラが見える
- 撮りたい・撮られたいを事前に伝えられる
- 登録しておくと、同じイベントに行く人から見つけてもらいやすい

避ける方向性:

- ビジネス向けSaaSっぽい表現
- 抽象的すぎるマーケティングコピー
- 「最高の体験を提供します」のような中身の薄い表現
- コスプレ文化に合わない硬い言葉

## Data and Modeling Principles

データ設計では、以下を考慮してください。

- 1人が1イベントで複数キャラを登録できる
- 参加種別は、コスプレ / カメラマン / 一般参加 / 未定 などを扱える
- カメラマンは撮りたい作品・キャラを複数登録できる
- 作品名・キャラ名は検索・フィルタしやすい形にする
- 表示名、Xユーザー名、プロフィールURL、ポートフォリオURLを扱える
- 公開情報と非公開情報を分ける
- 管理者だけが見られる情報を明確に分ける

DB変更が必要な場合は、既存データへの影響を説明してから実装してください。
既存データを壊す可能性がある変更は勝手に行わないでください。

## Auth and Permission

認証・権限周りは慎重に扱ってください。

- Xログインはなりすまし防止のため重要
- 未ログインでも閲覧できる範囲を明確にする
- 登録・編集・削除は本人のみ可能にする
- 管理者専用機能は一般ユーザーに見せない
- テスト中の分析機能や管理機能は、運営だけが見られるようにする

## Analytics

運営改善のため、ユーザー行動を分析できる設計を検討してください。

知りたい指標:

- イベントページ閲覧数
- 参加表明ページ閲覧数
- 登録フォーム開始数
- 登録完了数
- フォーム離脱箇所
- X共有クリック数
- OGP経由の流入
- 滞在時間
- 参加表明カードのクリック

ただし、初期実装では過剰な分析基盤を作らず、運営だけが確認できる小さな実装から始めてください。

## Implementation Rules

実装時は以下を守ってください。

- 既存のコードスタイルに合わせる
- 不要な大規模リファクタリングをしない
- 変更範囲を小さくする
- 既存機能を壊さない
- UI変更時はスマホ表示を必ず考慮する
- 型エラー、lintエラー、ビルドエラーを残さない
- 重要な仕様判断はコメントまたはドキュメントに残す
- 仮実装やダミーデータは、本番コードに混ぜない
- 環境変数やシークレットをコードに直書きしない

## Before Editing Code

コードを変更する前に、以下を確認してください。

1. 関連ファイルを読む
2. 既存の実装方針を把握する
3. 影響範囲を確認する
4. 最小変更で実装する
5. 必要ならTODOを分ける

## Response Style for Claude Code

Claude Codeが作業する時は、以下の形式で説明してください。

1. 何を変更したか
2. なぜその変更が必要か
3. 影響範囲
4. 確認方法
5. 残課題

大きな設計変更が必要な場合は、いきなり実装せず、先に方針を提案してください。

## Acceptance Criteria

実装完了時は、以下を満たしているか確認してください。

- スマホで見やすい
- 参加者・作品・キャラが探しやすい
- CTAが分かりやすい
- 登録フォームが重すぎない
- SNS共有時に見栄えが良い
- 管理者専用機能が一般ユーザーに見えていない
- 既存機能が壊れていない
- ビルド・lint・型チェックが通る

迷った時は、機能の多さよりも「イベント前に人を見つけやすいか」を優先してください。

登録項目を増やす場合は、入力負荷に見合うだけの見返りがあるかを確認してください。

UIを派手にする場合は、作品名・キャラ名・参加時間帯の読みやすさを犠牲にしないでください。
